## 文档概述

## 各文档详细说明

### 1. spec.md — 功能规格说明书

**定位**：项目的需求源头，所有设计和开发工作的依据。

**核心内容**：

- **用户故事（User Stories）**：按角色分为三类
  - 学生买家（US-B01 ~ US-B07）：搜索商品、查看详情、在线聊天、微信支付、确认收货、评价卖家、查看信誉
  - 学生卖家（US-S01 ~ US-S06）：发布商品、管理商品、聊天沟通、收款结算、评价买家、信誉积累
  - 通用用户（US-C01 ~ US-C04）：实名认证、密码重置、隐私保护、投诉工单
- **功能列表**：将用户故事细化为可执行的功能步骤
  - 用户认证模块（F-01 注册认证、F-02 登录、F-03 密码重置）
  - 商品浏览与搜索模块
  - 商品发布与管理模块
  - 在线聊天模块
  - 支付与交易模块
  - 评价模块
  - 投诉工单模块
- **验收标准**：每个功能的通过条件
- **边界条件**：异常场景处理规则
- **非功能性需求**：性能、安全、可用性指标

**关键设计决策**：
- 金额统一使用"分"为单位，避免浮点精度问题
- 学号/手机号等敏感信息脱敏展示
- 支付采用平台担保模式（买家付款 → 平台托管 → 确认收货 → 结算给卖家）

### 2. 系统架构设计与技术选型.md — 系统架构设计

**定位**：项目的技术蓝图，定义整体架构风格和技术栈。

**核心内容**：

- **架构风格**：分层单体 + 模块化（Modular Monolith）
  - 初期单体部署降低运维成本
  - 模块边界清晰，后期可平滑拆分为微服务
- **技术选型**：
  | 类别 | 技术 | 版本 |
  |------|------|------|
  | 语言 | Java | 17 LTS |
  | 框架 | Spring Boot | 3.2.x |
  | ORM | MyBatis-Plus | 3.5.x |
  | 数据库 | MySQL | 8.0 |
  | 缓存 | Redis | 7.x |
  | 认证 | Spring Security + JWT | — |
  | 实时通信 | WebSocket (STOMP) | — |
  | 消息队列 | RabbitMQ | 3.x |
  | 对象存储 | Alibaba Cloud OSS | — |
- **项目结构**：10 个 Maven Module
  - `campus2hand-common`：公共模块（异常、工具类、配置）
  - `campus2hand-auth`：认证模块
  - `campus2hand-user`：用户模块
  - `campus2hand-product`：商品模块
  - `campus2hand-order`：订单/支付模块
  - `campus2hand-chat`：聊天模块
  - `campus2hand-review`：评价模块
  - `campus2hand-dispute`：投诉工单模块
  - `campus2hand-notification`：通知模块
  - `campus2hand-job`：定时任务模块
  - `campus2hand-server`：启动模块
- **分层架构**：Controller → Service → Repository 三层结构
- **部署架构**：Nginx 反向代理 + Spring Boot 应用 + MySQL + Redis + RabbitMQ

**关键设计决策**：
- 采用 Modular Monolith 而非微服务，匹配当前团队规模
- 模块间通过接口（Provider 模式）解耦，避免循环依赖
- 认证采用 Spring Security + JWT 无状态方案

### 3. 数据库设计与初始化.md — 数据库设计

**定位**：数据库的蓝图，定义所有表结构和索引策略。

**核心内容**：

- **ER 实体关系图**：可视化展示 8 张核心表的关系
- **表清单**（8 张核心表）：
  | 表名 | 说明 | 核心字段 |
  |------|------|----------|
  | `users` | 用户表 | student_id, password_hash, nickname, campus_email, reputation_score |
  | `products` | 商品表 | seller_id, title, price, condition, category, status |
  | `orders` | 订单表 | order_no, buyer_id, seller_id, product_id, amount, status |
  | `payments` | 支付表 | order_id, wx_trans_id, amount, status |
  | `conversations` | 会话表 | buyer_id, seller_id, product_id |
  | `messages` | 消息表 | conversation_id, sender_id, content, message_type |
  | `reviews` | 评价表 | reviewer_id, reviewee_id, order_id, rating, content |
  | `disputes` | 投诉工单表 | order_id, creator_id, type, status |
  | `dispute_comments` | 工单留言表 | dispute_id, sender_id, content |
  | `participant_reads` | 已读状态表 | conversation_id, user_id, last_read_msg_id |
- **DDL 建表脚本**：完整的 CREATE TABLE 语句
- **索引设计**：主键、唯一索引、普通索引、全文索引
- **初始化数据**：校区白名单、分类枚举等种子数据
- **数据库配置**：连接池、字符集、时区设置

**关键设计决策**：
- 字符集 utf8mb4 + utf8mb4_unicode_ci 排序规则
- 金额以 BIGINT 存储"分"，避免浮点精度问题
- 手机号 AES-256 加密存储
- 密码 bcrypt 哈希存储
- 软删除策略（is_deleted 标记）

### 4. API接口设计.md — API 接口规范

**定位**：前后端交互的契约文档，定义所有接口的请求/响应格式。

**核心内容**：

- **通用规范**：
  - 统一响应格式 `ApiResponse<T>`：code + message + data + timestamp + traceId
  - 分页响应 `PageResponse<T>`：page + size + total + pages + list
  - 错误码枚举：1001（参数校验失败）~ 5000（服务器内部错误）
  - 认证方式：Bearer Token (JWT HS256)
  - 基础路径：`/api/v1`
- **接口清单**（按模块）：
  | 模块 | 接口数 | 主要接口 |
  |------|--------|----------|
  | 认证模块 | 6 | 注册、登录、邮箱验证、忘记密码、重置密码、刷新 Token |
  | 用户模块 | 4 | 个人信息、更新信息、修改密码、用户公开信息 |
  | 商品模块 | 8 | 搜索、详情、发布、更新、下架、标记售出、我的商品、卖家评价 |
  | 订单模块 | 7 | 创建订单、支付、详情、确认收货、申请退款、处理退款、订单列表 |
  | 聊天模块 | 6 | 会话列表、历史消息、发送消息（WebSocket）、上传图片、标记已读 |
  | 评价模块 | 2 | 提交评价、用户信誉查询 |
  | 投诉工单 | 4 | 创建工单、详情、追加留言、工单列表 |
  | 微信回调 | 1 | 支付结果回调通知 |
- **WebSocket 协议**：消息帧格式（SEND / RECEIVE / ACK）、心跳机制

**关键设计决策**：
- 所有接口统一 `/api/v1` 前缀，便于版本管理
- 分页参数统一为 page + size，默认 20 条/页
- 时间格式统一 ISO 8601 UTC
- 金额单位统一为"分"

### 5. tasks.md — 开发任务分解

**定位**：项目管理的执行计划，将功能需求拆解为可分配、可追踪的开发任务。

**核心内容**：

- **优先级定义**：P0（MVP 必需）→ P1（V1.0 必需）→ P2（V1.1 迭代）→ P3（后续版本）
- **11 个开发阶段**：
  | 阶段 | 内容 | 任务数 |
  |------|------|--------|
  | 阶段 0 | 项目初始化与环境搭建 | 16 |
  | 阶段 1 | 用户认证模块 | 12 |
  | 阶段 2 | 商品发布与管理模块 | 11 |
  | 阶段 3 | 商品浏览与搜索模块 | 7 |
  | 阶段 4 | 在线聊天模块 | 9 |
  | 阶段 5 | 支付与交易模块 | 12 |
  | 阶段 6 | 评价模块 | 6 |
  | 阶段 7 | 投诉工单模块 | 7 |
  | 阶段 8 | 个人中心模块 | 5 |
  | 阶段 9 | 管理后台 | 8 |
  | 阶段 10 | 部署与运维 | 7 |
- **依赖关系**：每个任务标注前置依赖，确保开发顺序合理
- **关联追踪**：每个任务关联具体的用户故事（US 编号）

**关键设计决策**：
- 阶段 0~5 为 MVP 核心流程，优先完成
- 认证模块必须最先完成，作为其他模块的前置依赖
- 前后端任务并行标注，便于全栈协作

### 6. checklist.md — 验收清单

**定位**：质量保证的最后一道关卡，交付前逐项验证。

**核心内容**：

- **按模块组织的检查项**：
  | 模块 | 检查项数 | 覆盖范围 |
  |------|----------|----------|
  | 用户认证 | 25 | 注册流程、登录、密码重置 |
  | 商品发布与管理 | 19 | 发布校验、管理操作 |
  | 商品浏览与搜索 | 12 | 搜索、筛选、排序、分页 |
  | 在线聊天 | 15 | WebSocket 连接、消息收发、已读 |
  | 支付与交易 | 22 | 下单、支付、回调、退款 |
  | 评价 | 10 | 提交评价、信誉计算 |
  | 投诉工单 | 12 | 创建、处理、留言 |
  | 个人中心 | 8 | 信息展示、脱敏、修改 |
  | 管理后台 | 10 | 审核、统计、用户管理 |
  | 非功能性 | 12 | 性能、安全、兼容性 |
- **检查状态**：☐ 未检查 / ✅ 通过 / ❌ 未通过
- **备注栏**：记录异常情况和处理方式

**关键设计决策**：
- 覆盖正常流程 + 异常边界条件
- 包含并发场景验证（如并发注册、并发购买）
- 包含安全验证（SQL 注入、XSS、敏感信息泄露）

## 维护说明

- 所有文档版本均为 v1.0，日期 2026-05-13
- 当需求变更时，应先更新 `spec.md`，再同步更新下游设计文档
- 代码实现应与文档保持一致，发现不一致时应及时修正文档或代码
- 建议在每次迭代结束后更新 `tasks.md` 中的任务状态
- 交付前必须完整走查 `checklist.md` 中的所有检查项
