### 1.2 架构风格

采用 **分层单体 + 模块化** 架构，按业务领域垂直切分为独立 Maven Module，水平方向遵循 Controller → Service → Repository 三层结构。初期以单体应用部署，模块边界清晰，后期可平滑拆分为微服务。

| 维度       | 选型                  | 理由                                       |
|------------|-----------------------|--------------------------------------------|
| 架构风格   | Modular Monolith      | 团队规模较小，单体降低运维成本，模块化保证可拆分 |
| 通信方式   | 模块内方法调用 + HTTP | 内部模块直接调用，外部通过 REST API         |
| 实时通信   | WebSocket (STOMP)     | 聊天消息推送，心跳保活                      |
| 异步任务   | Spring @Async + RabbitMQ | 支付回调、结算、审核等异步解耦          |

## 2. 技术选型

### 2.1 核心框架

| 类别         | 技术                        | 版本      | 选型理由                                        |
|--------------|-----------------------------|-----------|--------------------------------------------------|
| 语言         | Java                        | 17 LTS    | 长期支持版本，生态成熟，ZGC 低延迟              |
| 基础框架     | Spring Boot                 | 3.2.x     | 自动配置、起步依赖、Actuator 监控               |
| Web 框架     | Spring MVC (内嵌 Tomcat)    | —         | RESTful API，与 Spring 生态无缝集成              |
| 持久层       | MyBatis-Plus                | 3.5.x     | 简化 CRUD，分页插件，条件构造器，代码生成器     |
| 缓存         | Spring Cache + Redis        | —         | 注解式缓存，Redis 存储分布式 Session / 锁       |
| 安全         | Spring Security + JWT       | —         | 认证授权，bcrypt 密码编码，方法级权限控制       |
| 校验         | Hibernate Validator         | —         | Bean Validation，DTO 参数校验                   |
| 工具库       | Lombok + Hutool             | —         | 减少样板代码 + 通用工具集                       |

### 2.2 数据存储

| 类别         | 技术                      | 版本     | 用途                                 |
|--------------|---------------------------|----------|--------------------------------------|
| 关系数据库   | MySQL                     | 8.0.33+  | 核心业务数据（用户、商品、订单等）    |
| 缓存         | Redis                     | 7.x      | Session / Token / 分布式锁 / 热点数据 |
| 对象存储     | MinIO / 阿里云 OSS        | —        | 商品图片、聊天图片、缩略图           |
| 全文搜索     | MySQL FULLTEXT / Elasticsearch | —   | 商品标题+描述全文检索               |

### 2.3 消息与异步

| 类别         | 技术                      | 用途                                 |
|--------------|---------------------------|--------------------------------------|
| 消息队列     | RabbitMQ / Redis Stream   | 支付回调处理、结算异步、审核通知     |
| 定时任务     | Spring @Scheduled         | 订单超时取消、退款超时升级、补单查询 |
| 异步执行     | Spring @Async             | 邮件发送、内容审核、日志记录         |

### 2.4 监控与运维

| 类别         | 技术                      | 用途                                 |
|--------------|---------------------------|--------------------------------------|
| 指标监控     | Micrometer + Prometheus   | JVM 指标、接口 QPS/RT、业务指标      |
| 日志         | Logback + JSON 格式       | 结构化日志，traceId 全链路追踪       |
| 健康检查     | Spring Actuator           | /health /metrics /info 端点          |
| API 文档     | SpringDoc OpenAPI (Swagger) | 接口文档自动生成，调试测试         |
| 熔断降级     | Resilience4j              | 第三方服务（校园系统/微信支付）熔断  |

### 2.5 第三方集成

| 类别         | 技术                      | 用途                                 |
|--------------|---------------------------|--------------------------------------|
| 微信支付     | 微信支付 V3 API           | 统一下单、回调通知、退款、转账       |
| 校园系统     | HTTP Client (RestTemplate) | 学号+校园卡密码身份校验             |
| 邮件         | JavaMailSender (SMTP)     | 邮箱验证、密码重置邮件               |
| 内容审核     | 自建敏感词库 / 第三方API  | 商品标题+描述敏感词检测              |

## 4. 分层架构设计

### 4.1 分层调用关系

```
┌───────────────────────────────────────────┐
│  Controller 层                             │
│  - 接收 HTTP 请求                          │
│  - 参数校验 (@Valid)                       │
│  - 调用 Service                            │
│  - 返回统一响应 ApiResponse<T>             │
└──────────────┬────────────────────────────┘
               │
┌──────────────▼────────────────────────────┐
│  Service 层                                │
│  - 业务逻辑编排                             │
│  - 事务管理 (@Transactional)               │
│  - 调用第三方服务                            │
│  - 缓存操作                                 │
└──────────────┬────────────────────────────┘
               │
┌──────────────▼────────────────────────────┐
│  Repository (Mapper) 层                    │
│  - MyBatis-Plus BaseMapper                 │
│  - 自定义 SQL (XML / Annotation)           │
│  - 数据库交互                               │
└───────────────────────────────────────────┘
```

### 4.2 请求-响应规范

**统一响应体：**

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "timestamp": "2026-05-13T08:00:00Z",
  "traceId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

**统一错误码（部分）：**

| code | 说明                     |
|------|--------------------------|
| 0    | 成功                     |
| 1001 | 参数校验失败              |
| 1002 | 资源不存在               |
| 1003 | 无权限                   |
| 1004 | Token 过期或无效          |
| 1005 | 请求过于频繁              |
| 2001 | 用户未实名认证            |
| 2002 | 账户已锁定               |
| 3001 | 支付失败                  |
| 3002 | 订单状态不合法            |
| 5000 | 服务器内部错误            |

### 4.3 分页规范

**请求参数：**

| 字段 | 类型   | 默认值 | 说明               |
|------|--------|--------|--------------------|
| page | int    | 1      | 页码（从 1 开始）   |
| size | int    | 20     | 每页条数（最大 50）  |

**响应格式：**

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "page": 1,
    "size": 20,
    "total": 150,
    "pages": 8,
    "list": []
  }
}
```

使用 MyBatis-Plus 的 `Page<T>` 对象自动转换。

## 7. 关键设计决策

### 7.1 认证与授权

| 决策点       | 方案                          | 理由                                      |
|--------------|-------------------------------|-------------------------------------------|
| Token 格式   | JWT（HS256 签名）             | 无状态，适合水平扩展                        |
| Token 有效期 | Access: 2h; Refresh: 7d      | 短有效期防止泄露风险，长 Refresh 提升体验   |
| 密码存储     | bcrypt (cost=10)              | 抗彩虹表，计算成本足够高                    |
| 认证流程     | Spring Security Filter Chain  | 成熟框架，与 Spring 生态集成                |
| 接口权限     | @PreAuthorize + 方法级注解     | 声明式权限控制，可读性好                    |

### 7.2 分布式锁

| 场景             | 锁 Key                     | TTL    | 理由                |
|------------------|----------------------------|--------|---------------------|
| 并发购买         | `lock:order:{productId}`   | 10s    | 防止同一商品被多人拍 |
| 并发注册         | `lock:register:{studentId}`| 5s     | 防止学号重复注册    |
| 退款/结算        | `lock:settle:{orderId}`    | 30s    | 防止重复结算        |

使用 **Redisson** 实现 Redis 分布式锁，自动续期 + 看门狗机制。

### 7.3 金额处理

- **数据库**：`BIGINT` 存储，单位为**分**（避免浮点精度问题）
- **API 传输**：JSON 中金额字段为 `int` 类型，单位分
- **前端展示**：除以 100 后格式化为元显示
- **所有金额计算**：使用 `BigInteger` 或 `long` 运算，严禁 `double`

### 7.4 内容审核策略

| 阶段       | 审核方式         | 说明                               |
|------------|------------------|------------------------------------|
| 发布时     | 同步敏感词检测   | 内置词库 + 正则，≤ 1s              |
| 发布后     | 异步图片审核     | 第三方 API / 自建模型，≤ 2h      |
| 审核失败   | 通知用户+原因    | 可修改后重新提交                    |

### 7.5 聊天消息可靠性

- **消息发送**：客户端 → 服务端 → 落库 → 返回 ACK → 推送给对方
- **去重机制**：基于 `client_msg_id` 唯一约束防止重复
- **离线消息**：对方不在线时消息存库，上线后批量拉取
- **WebSocket 心跳**：30s 间隔 PING/PONG，3 次失败断开重连

### 7.6 微信支付回调可靠性

```
微信回调 → 验签 → 解密 → 更新订单 → 返回 200
                ↓ (失败)
         记录告警日志
                ↓
    定时主动查询(30s) → 对比状态 → 补单
```

- 验签**必须**通过才处理
- 回调重复：`order_no` 幂等处理（数据库唯一约束）
- 金额校验：回调金额与订单金额不一致 → 拒绝 + 告警

### 7.7 敏感数据保护

| 数据       | 存储方式               | API 返回       | 日志    |
|------------|------------------------|----------------|---------|
| 密码       | bcrypt 哈希            | 不返回          | 不打印  |
| 学号       | 明文（认证需要）        | 脱敏 20****01   | 脱敏    |
| 手机号     | AES-256 加密存储        | 脱敏 138****    | 脱敏    |
| 校园卡密码 | 不存储，仅转发校园系统  | 不返回          | 不打印  |
| Token      | Redis（服务端校验）     | 一次性返回       | 不打印  |
